# Agent Collaboration Guide

이 문서는 `moba-esports-manager-lite`에서 Codex와 Claude Code를 병행 사용할 때의 공통 운영 규칙입니다.
두 에이전트는 이 파일을 먼저 읽고, 같은 이슈를 동시에 다른 방향으로 수정하지 않도록 합니다.

## 역할 분리

### Codex

- GitHub 이슈, 라벨, 마일스톤 관리
- 이슈 본문 구체화와 장기 구현 계획 정리
- `md` 문서, 개발 규약, 디자인 규약, 핸드오프 문서 관리
- 구현 결과의 QA, 테스트 체크, 구조 리뷰
- 화면 관찰, 스크린샷 검증, 사용자 피드백 정리

### Claude Code

- 매치엔진 세부 구현
- 경기 타임라인 생성기, KDA/오브젝트 동기화, BO3/BO5 흐름 구현
- 도메인 로직과 단위 테스트 작성
- UI와 도메인 로직의 경계가 필요한 경우 구현 초안 제안
- 스펙이 모호한 경우 임의 구현 대신 질문 목록 작성

## 작업 흐름

1. 사용자가 처리할 이슈나 작업 범위를 정합니다.
2. Codex가 이슈 본문 또는 `md` 스펙을 구체화합니다.
3. Claude Code는 해당 스펙을 입력으로 받아 구현합니다.
4. Claude Code는 아래 핸드오프 양식으로 작업 결과를 전달합니다.
5. Codex는 변경 범위, 테스트, 이슈 충족 여부를 리뷰합니다.
6. 사용자가 최종 확인한 뒤 커밋, 푸시, 이슈 닫기 순서로 진행합니다.

## 충돌 방지 규칙

- 같은 이슈를 두 에이전트가 동시에 직접 구현하지 않습니다.
- 같은 파일을 동시에 수정하지 않습니다.
- 한 에이전트가 구현 중이면 다른 에이전트는 리뷰, 스펙 정리, 질문 정리만 수행합니다.
- 작업 시작 전과 종료 전에는 `git status --short`로 변경 상태를 확인합니다.
- 이미 존재하는 사용자 또는 다른 에이전트의 변경을 되돌리지 않습니다.
- 대규모 작업은 가능한 한 이슈 단위 또는 하위 단계 단위로 작게 나눕니다.

## 핸드오프 양식

Claude Code가 구현을 마치면 다음 형식으로 Codex와 사용자에게 전달합니다.

```md
작업 이슈:
수정 파일:
구현 요약:
실행한 테스트:
남은 TODO:
걱정되는 부분:
```

Codex가 Claude Code에게 구현 스펙을 넘길 때는 다음 내용을 포함합니다.

```md
작업 이슈:
목표:
구현 범위:
수정 대상 파일/폴더:
변경하지 말아야 할 범위:
필수 규칙:
테스트 계획:
완료 기준:
질문이 필요한 지점:
```

## 매치엔진 작업 기준

- `src/domain/live-match/`는 경기 프레젠테이션 모델, 밴픽, 타임라인, 스탯 동기화 같은 순수 로직을 우선 배치합니다.
- `src/features/live-match/`는 화면 컴포넌트와 표시 로직을 담당합니다.
- 도메인 로직은 가능한 한 UI에 의존하지 않게 작성합니다.
- 경기 결과를 두 번 시뮬레이션하지 않도록, 나중에는 결과 생성과 화면 재생 단계를 분리합니다.
- BO3/BO5, 피어리스 밴픽, KDA/오브젝트 동기화는 단계별로 확장합니다.

## 테스트 원칙

- Windows PowerShell에서는 `npm.cmd`를 사용합니다.
- 도메인 로직 변경은 관련 unit test를 함께 작성하거나 갱신합니다.
- UI 연결 변경은 가능하면 브라우저 smoke check 또는 integration test로 확인합니다.
- 전체 테스트가 실패할 경우, 해당 실패가 현재 작업과 직접 관련 있는지 분리해서 기록합니다.

## 커밋 전 체크

- `git status --short`
- `git diff --check`
- 관련 테스트 실행
- 필요 시 `npm.cmd run build`
- GitHub 이슈의 완료 조건과 실제 변경 내용 비교
