# 리스크와 AI 협업 흐름

## 1. 주요 리스크

### 범위 리스크

매니저 게임은 기능이 빠르게 커짐

대회, 선수, 계약, 저장, 이벤트, 밴픽, 장기 커리어가 모두 서로 연결되므로 구현 단위를 계속 쪼개야 함

### 밸런싱 리스크

기능이 많아도 선택이 결과에 영향을 주지 않으면 게임성이 약해짐

전략, 훈련, 선발, 밴픽 점수가 실제 결과에 반영되는지 반복 검증 필요

### 데이터 리스크

실제 LoL/LCK 데이터는 시즌마다 바뀜

개인 프로젝트에서는 실제 명칭을 사용하되, 장기적으로는 데이터 갱신 비용을 고려해야 함

### 저장 리스크

MongoDB 연결 후 상태 구조와 저장 문서 구조가 어긋날 수 있음

`version` 필드, migration, 저장 시점 설계 필요

### UI 리스크

정보량이 많은 게임이라 화면이 쉽게 복잡해짐

16:9 고정 프레임, 패널 내부 스크롤, 밀도 있는 카드 구성이 중요

### 장기 커리어 리스크

20시즌을 목표로 하면 선수 풀, 성장, 은퇴, 군 입대, 유망주 생성, 팀 전력 밸런스가 무너지지 않아야 함

## 2. AI 활용 방식

AI가 잘 맡을 수 있는 일:

- 기능을 작은 단위로 쪼개기
- 구현 전 질문 목록 만들기
- 도메인 타입 설계
- 일정/대회 포맷 정리
- 선수/팀/이벤트 데이터 초안 생성
- 테스트 케이스 생성
- PR 설명 작성
- Wiki 문서 초안 작성
- 16:9 UI 검증용 체크리스트 작성

사용자가 직접 판단해야 할 일:

- 게임의 재미 방향
- UI 취향
- 실제 LoL/LCK 포맷을 얼마나 반영할지
- 밸런싱 감각
- 공개 범위와 에셋 사용 범위
- GitHub PR 최종 내용
- Wiki/회고 최종 문장

## 3. 권장 작업 흐름

```text
문서 읽기
-> 현재 코드 확인
-> 구현 범위 질문
-> 사용자 컨펌
-> 도메인 로직 구현
-> UI 연결
-> 테스트
-> 16:9 스크린샷
-> 체크리스트 갱신
```

## 4. AI 요청 패턴

구현 전:

```text
이번 구현 주제는 [기능명]이야.
먼저 현재 코드와 관련 문서를 읽고, 구현 범위와 내가 결정해야 할 질문을 정리해줘.
내가 컨펌하면 구현을 시작해줘.
```

구현 후:

```text
방금 구현한 내용, 바뀐 파일, 검증 결과, 남은 작업을 정리해줘.
UI 변경이 있으면 16:9 스크린샷도 저장해줘.
```

검증 요청:

```text
이 기능이 실제 게임 흐름에서 잘 동작하는지 단위/통합/시스템/인수 관점으로 검증해줘.
실패하거나 부족한 부분은 수정해줘.
```

## 5. 작업 단위 쪼개기 기준

좋은 작업 단위:

- 하나의 대회 엔진
- 하나의 화면
- 하나의 상태 전환
- 하나의 테스트 범위
- 하나의 저장 흐름

피해야 할 작업 단위:

- "국제전 전부 구현"
- "저장 완성"
- "밸런싱 전부 처리"
- "UI 전체 예쁘게"

## 6. 완료 판단 체크포인트

기능 완료 전 확인:

- 코드가 현재 타입 구조와 맞는지
- 기존 사용자 결정을 뒤집지 않았는지
- 진행 흐름이 끊기지 않는지
- 테스트가 필요한 도메인 로직에 추가됐는지
- `npm run build`가 통과하는지
- UI가 16:9에서 깨지지 않는지
- 체크리스트가 최신인지

## 7. GitHub 관리

- 기능 단위 Issue 생성
- `feature/*` 브랜치에서 작업
- `dev`로 PR 생성
- PR에는 작업 내용, 검증 방법, 스크린샷, 한계점 작성
- Wiki에는 기획/기술/Agent workflow/회고 정리

## 8. 장기 운영 메모

과제 제출과 별개로 이 프로젝트는 장기 개인 프로젝트 지향

단기 목표는 1시즌 완주

중기 목표는 3시즌 이상 안정 플레이

장기 목표는 20시즌 커리어와 밸런싱
