## 1. 현재 전제

기준일: 2026-06-06

## 4. 리스크

가장 큰 리스크는 구현량보다 밸런싱

기능이 있어도 선택이 결과에 영향을 준다는 감각이 약하면 매니저 게임으로서 재미가 떨어짐

두 번째 리스크는 데이터 확장

선수, 팀, 챔피언, 국제전 상대가 늘어날수록 테스트와 검수가 필요

세 번째 리스크는 저장 운영화

MongoDB 연결 후에도 프론트엔드 상태와 서버 저장 데이터가 어긋나지 않게 schemaVersion, migration, save timing, ownerId 정책 설계 필요

## 5. 현실적인 개발 순서

1. MSI 완주
2. LCK 후반 시즌 완주
3. Worlds 완주
4. 시즌 요약과 다음 시즌
5. 저장 자동화와 로그인/사용자별 저장 준비
6. 장기 이벤트
7. 밸런싱

저장 구조는 1차 구현됐으므로, 다음 저장 작업은 자동 저장 시점과 배포 전 DB 분리 정책을 중심으로 진행
