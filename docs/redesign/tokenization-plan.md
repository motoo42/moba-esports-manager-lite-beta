# 토큰화 패스 계획 (승인 후 착수)

> 목적: 하드코딩된 색을 토큰 참조로 정리해, "토큰값만 바꾸면 전 화면이 리스킨되는" 상태를 만든다.
> **이 작업은 색을 바꾸지 않는다. 보이는 결과는 동일하고, 배선(토큰 사용)만 정리한다.**

## 비목표 (이번에 하지 않는 것)

- 색감/디자인 변경 ❌ (그건 토큰화 끝난 뒤, 방향 정하고)
- 마크업/요소 배치 변경 ❌ (방법 2, 별도)
- 라우팅·네비·핸들러·도메인 변경 ❌
- live-match 손대기 ❌ (다크 전용 예외)

## 단계

### Phase A — 기존 토큰값의 복붙 치환 (기계적, 저위험)
하드코딩 hex 중 **기존 토큰과 값이 같은 것**을 `var(--token)`으로 치환.
- 예: `#9aa7bd`→`var(--muted)`, `#0d1424`→`var(--surface-sunken)`, `#111827`→`var(--color-button-bg)`, `#303a52`→`var(--color-pill-border)`, `#cfd7e6`→`var(--color-menu-button-text)`, `#2a3245`→`var(--color-border)`, `#e8edf6`→`var(--text)`.
- 값이 동일하므로 **다크 모드 픽셀 변화 0** (검증 쉬움).

### Phase B — 신규 토큰 도입 후 치환
토큰이 없는데 자주 쓰인 색을 의미 토큰으로 승격 후 치환.
- 후보: `#253048`(67), `#9f5cff`(29), `#171436`(13), 그 외 빈도 높은 잔여색.
- 다크값을 그대로 토큰화(픽셀 변화 0), 라이트값은 §라이트 단계에서 정의.

### Phase C — 라이트 땜빵 정리
베이스가 토큰을 쓰게 되면 불필요해진 `:root[data-theme="light"] .클래스` 626개 중 **중복/무용한 것 제거**.
- 토큰 재정의로 자동 상속되는지 라이트 모드에서 화면 확인하며 안전하게 삭제.
- 라이트 전용 미세조정이 진짜 필요한 소수만 남김.

## 진행 방식 — 파일럿 먼저

1. **파일럿 범위**: AppShell(사이드바/탑바) + 홈(main-dashboard) 관련 클래스만 Phase A·C 적용.
2. 빌드/테스트/가드레일 통과 + 다크·라이트 육안 확인.
3. **결과 보고 → 승인** → 나머지 화면으로 확대(섹션 단위로 점진).

## 검증 (각 단계마다)

- `npm.cmd run build` 통과
- `npm.cmd test` — 기존 통과 테스트 유지 (사전 실패 2건 제외)
- 토큰 가드레일(`DARK_SURFACE_HEXES`) 회귀 없음(↓ 방향)
- 다크/라이트 모드 육안 비교: 픽셀 변화 없어야 정상(Phase A·B), 라이트는 동등하거나 개선(Phase C)

## 리스크 / 롤백

- 위험 낮음(값 동일 치환). 화면별·섹션별 커밋으로 쪼개 문제 시 부분 롤백.
- 커밋/푸시는 사용자 확인 후 진행. (origin/dev + beta/main 이중 푸시 규칙 준수)
