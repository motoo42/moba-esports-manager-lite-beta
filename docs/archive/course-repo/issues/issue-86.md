## 목표
라이브 매치 재생 중, 선수의 아이템이 경기 시작 시점부터 시간 흐름에 따라 점진적으로 채워지고 업그레이드되도록 한다. 현재는 경기 내내 완성형 아이템 세트가 고정 표시되어 비현실적이다(#34 step 6 기준).

## 핵심 아이디어
골드 모델은 그대로 두고, **"시점 T의 보유 아이템 = 빌드 순서대로 현재 골드로 살 수 있는 prefix"** 로 도출한다.

- 살 수 있을 때 구매하므로 `골드 >= 구매 합계`가 자동 보장된다.
- 남는 골드 = 현재 골드 − 구매 합계 < 다음 아이템 가격 → "표시 골드가 아이템 가격 총합보다 약간 큼"이 자연스럽게 성립한다.
- 재생이 이미 시점별 골드를 제공하므로, 시간에 따라 아이템이 차오르는 연출이 추가 비용 없이 따라온다.

## 1차 구현 범위 (옵션 A — 추천)
하위 컴포넌트는 모델링하지 않고, 완성 아이템이 구매 가능 시점에 슬롯에 등장한다.

- `matchItemPool`(약 50개)에 `cost`(완성가) 추가. 결정성/오프라인을 위해 하드코딩 권장(Data Dragon `item.json`의 `gold.total` 참고).
- 역할별 빌드 순서(구매 시퀀스) 정의. 신발은 빌드 초반. 기존 완성 세트를 순서화하고 신발 타이밍을 부여.
- 순수 함수 `getOwnedItemsAt(buildOrder, costs, gold)` → 보유 아이템 도출.
- 어댑터/스탯보드가 스냅샷의 시점별 골드로 아이템 슬롯을 산출(현재 정적 mock 대체).

## 후속 확장 (옵션 B)
"하위아이템 구매 → 점점 업그레이드"를 실제로 표현.

- 빌드트리 데이터(컴포넌트 `from`, 컴포넌트 가격, 조합 비용) 필요. 풀이 약 50개 → 약 150개 이상으로 커짐.
- 부분 빌드 로직(컴포넌트 단위 구매 → "조립 중" 표시 → 완성 시 교체).
- 컴포넌트/진행중 슬롯 렌더 UI.

## 비목표
- 실제 라이엇 가격·빌드트리를 100% 정밀 재현하지 않는다(1차는 근사).
- 챔피언/메타별 최적 빌드 추천 로직은 만들지 않는다.
- 아이템 능력치가 경기 결과 계산에 영향을 주지 않는다(연출 레이어 유지, #34 책임 분리 계승).

## 선행 이슈 / 의존 관계
- 선행: #34 매치엔진 화면 (step 5 스탯 fold의 시점별 골드, step 6 신발 슬롯 분리)
- 병행 가능: #72 선수 능력치 시스템 구체화
- 후속 연결: #71 개인 기록·데이터베이스·통계 시스템

## 데이터 모델 변경
- `MatchItem`에 `cost: number` 추가(옵션 B 시 `from`, `combineCost` 추가).
- 역할별 빌드 순서 데이터(아이템 id 시퀀스, 신발 위치 포함).
- (옵션 B) 컴포넌트 아이템 풀.

## UI/UX 변경
- 스탯보드 아이템 슬롯이 시점별 보유 아이템을 표시(시간에 따라 증가).
- (옵션 B) 조립 중 컴포넌트/진행 상태 시각화.

## 단계별 구현 계획
1. 아이템 가격 데이터 추가
2. 역할별 빌드 순서 정의
3. `getOwnedItemsAt` 순수 함수 + 단위 테스트
4. 어댑터/스탯보드 연결(시점별 아이템 산출)
5. 검증 / 스크린샷
6. (후속) 컴포넌트 업그레이드 모델링

## 테스트 계획
- 보유 아이템이 시간에 따라 단조 증가
- `골드 >= 구매 합계`, 남는 골드 < 다음 아이템 가격
- 빌드 순서 / 신발 슬롯 정합
- 결정적(시드 동일 → 동일 결과)

## 완료 기준 (옵션 A)
- [ ] 경기 초반엔 아이템이 적고, 시간이 지나며 채워진다
- [ ] 표시 골드가 구매 아이템 합계보다 항상 약간 크다
- [ ] 신발 슬롯이 빌드 순서에 맞게 채워진다
- [ ] 시점별 아이템 산출이 결정적이고 단조 증가한다
- [ ] 관련 단위 테스트 + 브라우저 smoke 통과

## 작업량 추정
- 옵션 A: 활성 약 3~4시간 / 난이도 中 (대부분 데이터 작업, 알고리즘 단순)
- 옵션 B: 추가 약 6~9시간 / 난이도 中上 (컴포넌트 데이터·로직·UI 각 1.5~2배)
