## 목표
경기 진행 버튼을 눌렀을 때 단순 대기 오버레이가 아니라, 밴픽/세트 진행/주요 장면/결과가 단계적으로 표시되는 매치엔진 화면을 구현한다.

초기 버전은 정교한 실시간 엔진이 아니라, 이미 계산된 경기 결과를 게임적으로 보여주는 **문자중계형 매치 진행 화면**으로 시작한다.

## 2026-06-15 구체화 방향
남은 구현 시간과 안정성을 고려해, #34의 1차 구현은 생동감 있는 실시간 전투 화면이 아니라 **자동 밴픽 → 문자중계 → 하단 선수 스탯 패널 → 세트/매치 결과**로 이어지는 전체화면 연출형 매치엔진으로 구체화한다.

- 실제 전투 애니메이션 대신 중앙에는 주요 사건 중심의 문자중계 피드를 보여준다.
- 하단에는 실제 LCK 중계 화면을 레퍼런스로 삼은 10인 선수 스탯 패널을 배치한다.
- 기존 match simulation이 결정한 승패/세트 결과를 source of truth로 사용한다.
- 매치 화면은 별도 승패 판정 엔진이 아니라, 이미 결정된 결과를 설득력 있게 재생하는 연출 레이어로 둔다.

## 핵심 화면 원칙
매치 진행은 기존 앱 쉘 안의 모달이나 작은 패널이 아니라, 아예 별도의 매치 전용 전체화면으로 이동해 진행한다.

- 좌측 사이드 메뉴와 상단 일반 메뉴를 숨긴다.
- 화면 전체를 매치 진행, 밴픽, 세트 로그, 선수 스탯, 결과 표시만을 위해 사용한다.
- 유저가 온전히 경기 자체에 집중할 수 있는 구조를 우선한다.
- 경기 종료 후에는 기존 경기 리뷰/결과 흐름으로 자연스럽게 복귀한다.

## 화면 흐름
1. 자동 밴픽 단계
2. 경기 로딩 / 세트 시작
3. 문자중계 + 하단 선수 스탯 갱신
4. 세트 결과 표시
5. BO3/BO5일 경우 다음 세트 반복
6. 매치 최종 결과 표시 후 기존 경기 결과 흐름으로 복귀

## 1차 구현 범위
### 자동 밴픽
- 1차 MVP에서는 유저 직접 조작이 아니라 자동 밴픽 연출로 구현한다.
- 실제 프로 경기 느낌을 살리기 위해 밴/픽 카드가 순서대로 채워지는 구조를 사용한다.
- 밴픽 순서는 실제 LoL 대회 흐름을 참고한다.
  - 1차 밴: Blue / Red / Blue / Red / Blue / Red
  - 1차 픽: Blue 1픽 → Red 1,2픽 → Blue 2,3픽 → Red 3픽
  - 2차 밴: Red / Blue / Red / Blue
  - 2차 픽: Red 4픽 → Blue 4,5픽 → Red 5픽
- 자동 생성 기준은 다음 정도로 제한한다.
  - 포지션별 챔피언 풀 사용
  - 같은 챔피언 중복 금지
  - 밴된 챔피언 픽 금지
  - 상대 핵심 선수/메타 점수 높은 챔피언을 밴할 확률 증가
  - 선수 포지션에 맞는 챔피언을 픽
- 유저가 직접 밴픽에 개입하는 기능은 후속 단계로 남긴다.

### 챔피언 / 아이템 데이터
- 챔피언 및 아이템 아이콘은 Riot Data Dragon 정적 리소스를 사용하는 방향으로 검토한다.
- Data Dragon은 정적 데이터/이미지 제공용이므로 API 키 없이 사용할 수 있다.
- MVP에서는 전체 챔피언/아이템 메타를 정교하게 구현하지 않고, 다음 정도로 제한한다.
  - 포지션별 8~12개 내외의 curated champion pool
  - 포지션별 대표 아이템 pool
  - 챔피언 스킬, 아이템 효과, 세부 상성 계산은 구현하지 않음
  - 아이콘과 이름을 사용해 화면의 실감 우선 확보

### 문자중계 구조
- 문자중계는 모든 사건을 보여주지 않고, 경기 흐름상 중요한 사건만 보여준다.
- 내부적으로는 1~2분 단위의 경기 타임라인을 계속 갱신한다.
- 화면에 보이는 문자중계는 `visible: true`인 주요 사건만 표시한다.
- 화면에 표시되지 않는 내부 이벤트도 K/D/A, 골드, 레벨, 아이템, 오브젝트 UI에는 반영된다.

예시 구조:

```ts
{
  time: 16.2,
  type: "dragon",
  teamId: "T1",
  visible: false,
  importance: "low"
}

{
  time: 24.6,
  type: "baron_steal",
  teamId: "T1",
  visible: true,
  importance: "high",
  message: "Oner의 스틸 성공! T1이 바론을 가져갑니다."
}
```

### 문자중계 타이밍
- 주요 사건은 고정 간격이 아니라 경기 구간별 가중 랜덤으로 배치한다.
- 사건이 항상 균등하게 흩어지면 어색하므로, 오브젝트/한타 주변에서는 1~2분 간격으로 연달아 발생할 수 있게 한다.
- 반대로 3~5분 정도 주요 중계가 없는 소강 상태도 허용한다.
- 경기 후반에는 이벤트 밀도를 조금 높인다.
- 마지막 2~3분 안에는 세트 종료로 이어지는 마무리 이벤트를 반드시 배치한다.

### 경기 시간 / 재생 시간
- 내부 경기 종료 시간은 세트마다 다르게 생성한다.
  - 일방적인 경기: 24~29분
  - 일반적인 경기: 30~36분
  - 접전 경기: 37~44분
  - 장기전: 45분 이상, 낮은 확률
- 화면 재생 시간은 내부 경기 시간을 압축해서 보여준다.
  - 기본 세트 재생: 약 45~60초
  - 빠른 진행: 약 25초
  - 즉시 결과: 현재 세트 바로 종료
- 예시 계산:

```ts
displayDurationMs = clamp(gameDurationMinutes * 1500, 38000, 65000)
```

### 하단 선수 스탯 패널
- 실제 LCK 중계 화면의 하단 선수 통계 패널을 레퍼런스로 삼는다.
- 단, 방송 그래픽을 그대로 복제하지 않고 프로젝트 고유의 매치 HUD로 제작한다.
- 좌측 5명 / 우측 5명 대칭 구조를 사용한다.
- 선수별 표시 정보:
  - 챔피언 아이콘
  - 선수명
  - 포지션
  - K / D / A
  - 레벨
  - 골드
  - 아이템 슬롯
- 문자중계 사이에도 내부 이벤트에 따라 KDA/골드/레벨/아이템이 갱신될 수 있다.
- 숫자가 바뀔 때는 가벼운 반짝임 또는 부드럽게 올라가는 애니메이션을 적용한다.

### 오브젝트 UI
- 주요 오브젝트는 상단 또는 상단 보조 바에 팀별로 표시한다.
- 반영 대상:
  - 드래곤
  - 전령
  - 바론
  - 장로
  - 타워
- 모든 오브젝트 획득을 문자중계하지는 않는다.
- 내부적으로는 모든 오브젝트를 기록하고 UI에 반영하되, 문자중계에는 다음 경우만 노출한다.
  - 첫 드래곤처럼 흐름을 여는 오브젝트
  - 3용/4용/영혼 등 중요한 드래곤 상황
  - 바론/장로
  - 오브젝트 앞 한타
  - 스틸
  - 오브젝트를 먹고도 한타를 패배한 장면
  - 마지막 진격과 연결되는 타워/억제기/넥서스 장면

## 2026-06-15 추가 구체화: 이벤트/스탯/챔피언 데이터 규칙
### 문자중계 팝업 / 표시 방식
- 문자중계 이벤트가 추가될 때 부드러운 메시지 팝업 효과를 적용한다.
- 메시지 표시 빈도는 최소 2단계 옵션을 제공한다.
  - 주요 상황: 기본 빈도, 경기 흐름상 의미 있는 사건 중심으로 표시
  - 핵심 상황: 더 낮은 빈도, 승부 흐름을 바꾸는 사건 위주로 표시
- 블루팀에게 유리한 결과는 파란색 색인이 붙은 메시지로 표시한다.
- 레드팀에게 유리한 결과는 붉은색 색인이 붙은 메시지로 표시한다.
- 양 팀에게 거의 평등하거나 중립적인 결과는 노란색 색인이 붙은 메시지로 표시한다.
- 메시지 색인은 단순 장식이 아니라 해당 이벤트의 실제 경기 영향도와 팀 유불리 판정을 반영해야 한다.

### 이벤트와 스탯 동기화
- 문자중계 이벤트는 K/D/A, 에픽 몬스터, 오브젝트 상태와 분명하게 연동되어야 한다.
- 킬 이벤트가 표시되면 관련 선수의 킬/데스/어시스트가 같은 타임라인에서 갱신되어야 한다.
- 드래곤, 전령, 바론, 장로, 타워 등 오브젝트 이벤트가 표시되면 상단 오브젝트 UI에도 같은 결과가 반영되어야 한다.
- 문자중계에 표시되지 않는 작은 이벤트도 내부 스탯에는 반영될 수 있지만, 빈도는 낮게 유지한다.

### 킬 이벤트 규칙
- 솔로킬과 교전킬을 구분한다.
  - 킬 발생 시 팀원 어시스트가 0개이면 솔로킬
  - 팀원 어시스트가 1~4개이면 교전킬
- 라인전 단계에서 발생한 솔로킬은 중요 이벤트로 취급한다.
- 킬/데스가 증가하지만 문자중계에는 표시되지 않는 영향도 낮은 소규모 교전도 허용한다.
  - 단, 이 빈도는 드물어야 한다.
  - 교전 규모도 작아야 한다.
- 타워 처형, 자살, 중립 오브젝트 처형처럼 킬 주체 없이 사망하는 상황은 v1에서는 제외한다.
- 따라서 팀 단위 수치는 항상 다음 관계를 만족해야 한다.
  - 블루팀 킬 = 레드팀 데스
  - 레드팀 킬 = 블루팀 데스
- 경기 전체 결과를 종합했을 때 팀 전체 킬 총합 대비 어시스트 총합 비율은 대략 2~3 사이로 맞춘다.

### 챔피언 데이터 1차 범위
- 궁극적으로는 전체 챔피언 데이터를 수집하고 싶지만, 1차 구현은 라인별 curated pool로 시작한다.
- 라인당 15~20개 챔피언 등록을 목표로 한다.
- 2개 이상의 라인에 설 수 있는 챔피언은 복수 라인 후보로 기록한다.
- 챔피언 데이터는 밴픽, 피어리스 누적 사용 카드, 선수별 챔피언 표시, 문자중계 문장에 함께 사용될 수 있어야 한다.
- 챔피언 스킬/아이템 효과까지 정밀 계산하는 것은 v1 범위가 아니며, 우선 포지션 적합성과 화면 실감을 확보한다.
