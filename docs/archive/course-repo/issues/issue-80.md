## Body

﻿## 목표
스토브리그 협상을 단일 오퍼 후 결과 확인 방식에서, 유저와 AI 구단이 서로 더 높은 제안을 낼 수 있는 경매식 협상 구조로 확장한다.

## 비목표
- v1에서 모든 FA 선수를 실시간 경매장처럼 운영하지 않는다.
- 과도한 메시지 폭증을 유발하는 반복 알림 구조를 만들지 않는다.
- 유저에게만 유리한 비대칭 협상 구조를 만들지 않는다.

## 선행 이슈 / 의존 관계
- 연계: #73 선수 영입 옵션 조항 추가
- 연계: 스토브리그 offer/resolve 구조
- 후속: AI 구단 성향, 예산, 로스터 니즈 고도화

## 핵심 규칙
- 최소 수락선과 확정 수락선을 분리한다.
- 최소 수락선 이상이면 선수가 계약을 고려하지만, 확정 수락선 미만이면 더 높은 제안이 나올 수 있다.
- 확정 수락선 이상이면 추가 오퍼 경쟁 없이 즉시 계약을 확정한다.
- 확정 수락선은 최소 수락선의 약 1.4배-1.6배 사이 금액으로 책정한다.
- 한 번 계산된 확정 수락선은 해당 협상 동안 고정한다.

## 공정한 추가 오퍼 원칙
유저가 AI보다 높은 제안을 했더라도, 그 제안이 확정 수락선 밑이라면 영입이 즉시 종료되지 않는다.

- 유저가 최고 제안자라도 확정 수락선 미만이면 AI 구단에게 추가 오퍼 기회가 제공될 수 있다.
- AI가 최고 제안자라도 확정 수락선 미만이면 유저에게 추가 오퍼 기회가 제공될 수 있다.
- 즉, 확정 수락선 아래에서는 최고 제안자가 누구인지와 무관하게 경쟁 구단이 반격할 수 있다.
- 확정 수락선 이상 제안만 경매를 즉시 종료한다.

## 협상 상태 모델
- pending: 제안 대기/결과 대기
- leading: 현재 최고 제안 보유
- outbid: 더 높은 제안에 밀림
- counter-window: 추가 오퍼 가능 기간
- accepted: 계약 확정
- rejected: 협상 결렬 또는 포기
- expired: 응답 기한 초과

## UI/UX 변경
- pending offer 카드에서 현재 최고 제안 여부 표시
- 더 높은 제안 발생 시 추가 오퍼 여부 확인
- 추가 오퍼 가능 시간 또는 일자 표시
- 포기/재제안/즉시 상향 제안 액션 제공
- 메시지함과 스토브리그 로그에 경쟁 상황을 명확히 기록

## AI 구단 제안 로직
- 팀 예산
- 해당 포지션 니즈
- 선수 등급/명성
- 기존 최고 제안 대비 상승 폭
- 확정 수락선과의 거리
- 구단별 공격성 또는 보수성 후보

## 단계별 구현 계획
1. offer 모델에 최소 수락선/확정 수락선/현재 최고 제안 상태 추가
2. AI 구단 제안 비교 로직 구현
3. 확정 수락선 이상 제안 즉시 계약 처리
4. 확정 수락선 미만 최고 제안에 대해 상대측 counter chance 부여
5. 유저 추가 오퍼 UI/메시지/로그 연결
6. 협상 종료/포기/기한 초과 처리

## 밸런스 고려사항
- 모든 협상이 무한 경매가 되지 않도록 최대 라운드 또는 기한을 둔다.
- 유저가 확정 수락선을 확인할 수 있을지, 추정만 가능할지 결정해야 한다.
- 경쟁이 치열한 고급 선수와 일반 선수의 입찰 빈도를 다르게 둔다.

## 테스트 계획
- 유저 제안이 확정 수락선 이상일 때 즉시 계약
- 유저가 최고 제안이지만 확정 수락선 미만일 때 AI counter 발생 가능
- AI가 최고 제안일 때 유저 counter 기회 발생
- 추가 오퍼 포기 시 경쟁 구단 계약 또는 결렬 처리
- 메시지/로그 중복 방지 테스트
